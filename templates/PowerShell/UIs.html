PowerShell - User Interfaces

Separating our User Interface with Functions

In this lesson we are going to learn how to keep our code nice and clean by seprating our code related to creating our UI from the rest of our code.

Before you begin:

In this lesson, we will be using a file named PowerShellUIs.ps1. This is a PowerShell script that contains our example code. It is not required that you use this file to follow along, but I think it will be helpful. You can find this file in the following github repo in a folder named examples: https://github.com/ScottVaiskauckas/PowerShell. If you have never used Github before, for full instructions on how to download this file, please see my page on Getting Started.

What's the point?

Why would I want to separate my UI code from the rest of my code? The main reason this is done in programming is to make your code easier to maintain and debug. If you know that all of the code related to one thing your program does is all in the same area, you know where to look when something breaks. In programming, these areas are called layers and in this case we are talking about the presentation layer.

Another reason for separating your UIs from the rest of your code is for a consistent experience. It is often possible to reuse the same function for presenting the output to the end user. This guarantees that the output has a similar feel each time it is generated. This may not matter much if you only have one person writing scripts, but if multiple people are involved it will make things feel more unified.

Wait a minute, PowerShell has a UI? Yeah, it does. When you're writing console programs the UI is the console. Just because it is text-based does not mean it is not a UI and that principles of good UI/UX design don't apply. If you have never used any legacy *nix programs, this may be a radical concept for you, but having suffered through some bad terminal app UI design choices myself, I think this is an often neglected area by administrators.

DisplayCollection()

So far, we have been handling creating our UIs by returning values from functions and passing them into a parameter of the Write-Host cmdlet. Consider Example4 from our Arraylist lesson. We can take all of the code related to creating the UI (lines 73-80) and place it inside of a function named DisplayCollection(). I have taken a few additional steps to refactor the code so it displays the collection type and iterates through all of the elements in the collection instead of just printing the first element.

Now whenever we need to generate a UI for our future demonstrations revolving around arrays and arraylists, we can just use this function.
